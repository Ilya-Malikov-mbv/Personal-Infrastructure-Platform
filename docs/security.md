# Безопасность

## Общие принципы

При разработке инфраструктуры особое внимание уделялось минимизации поверхности атаки и ограничению доступа к внутренним сервисам.

Основные принципы:

- минимально необходимые открытые порты;
- использование HTTPS;
- изоляция сервисов в Docker;
- доступ к внутренним сервисам только через Nginx;
- автоматическое обновление сертификатов.

---

# Сетевая архитектура

```
              Internet
                 │
             80 / 443
                 │
               Nginx
      ┌──────────┴──────────┐
      │                     │
 Grafana              Uptime Kuma
127.0.0.1:3000      127.0.0.1:3001

Prometheus
127.0.0.1:9090
```

Все внутренние сервисы прослушивают только интерфейс `127.0.0.1` и недоступны напрямую из Интернета.

---

# HTTPS

Для всех публичных сервисов используется HTTPS.

Сертификаты выпускаются через Let's Encrypt.

HTTP автоматически перенаправляется на HTTPS.

---

# Reverse Proxy

Пользователи взаимодействуют только с Nginx.

Это позволяет:

- скрыть реальные внутренние порты;
- централизованно управлять TLS;
- использовать единые правила безопасности;
- добавлять новые сервисы без изменения клиентской части.

---

# Docker

Каждый сервис работает в собственном контейнере.

Преимущества:

- изоляция зависимостей;
- независимое обновление;
- упрощенное восстановление после сбоев;
- ограничение влияния ошибок одного сервиса на остальные.

---

# Ограничение открытых портов

Во внешнюю сеть доступны только:

| Порт | Назначение |
| ---- | ---------- |
| 80   | HTTP       |
| 443  | HTTPS      |
| 22   | SSH        |

Prometheus, Grafana и Uptime Kuma доступны только через Nginx.

---

# SSH

Управление сервером осуществляется по SSH.

Рекомендуемые меры безопасности:

- использование SSH-ключей;
- отключение входа по паролю;
- запрет входа пользователю root;
- использование нестандартного порта (при необходимости).

---

# Fail2Ban

Для защиты SSH используется Fail2Ban.

Fail2Ban анализирует журналы аутентификации и автоматически блокирует IP-адреса, с которых фиксируется большое количество неудачных попыток входа.

Использование Fail2Ban позволяет значительно снизить эффективность атак типа brute force на SSH.

Типовой процесс работы:

```text
Неудачные попытки входа
           │
           ▼
    systemd-journal
           │
           ▼
        Fail2Ban
           │
           ▼
      Блокировка IP
```

# CI/CD

Автоматическое развертывание выполняется через GitHub Actions.

Workflow подключается к серверу по SSH и выполняет обновление проекта.

Это исключает необходимость ручного копирования файлов и снижает вероятность ошибок при деплое.

---

# Резервное копирование

Рекомендуется регулярно создавать резервные копии:

- конфигурации Nginx;
- Docker Compose;
- каталогов с данными Grafana;
- данных Uptime Kuma;
- конфигурации Prometheus.

---

# Дальнейшее развитие

Возможные направления усиления безопасности:

- настройка UFW;

- HTTP Security Headers;
- Content Security Policy;

- централизованный сбор логов;
